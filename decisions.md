# Decisions

Cross-project principles. These apply to everything built under `~/building/`. They are closed unless new material information surfaces.

Decisions are organized into three tiers by scope, not chronology. Numbers are stable — they never change, and other files reference them. Future decisions are added to whichever tier they belong to, maintaining sequential numbering across all tiers.

## Foundational Tenets — How to think about what you build

Principles about the relationship between the product and the person using it. They apply to every product, every feature, every design decision. An agent writing a PRD should internalize these. An agent reviewing an architecture should check against these.

| # | Decision | Rationale | Date |
|---|----------|-----------|------|
| 8 | Abstractions must be earned | Every abstraction is a bet on future change. For v1 of any project, the default is fewer abstractions. An abstraction earns its place when the cost of not having it is concrete and near-term. "We might need this later" is not earned. | 2026-03-08 |
| 15 | Non-functional requirements are product requirements | **Insight:** Agents build features that work correctly but ignore performance, security, and resilience unless explicitly told. No pagination, no indexing, no caching — the app works on day one and degrades from there. The agent optimizes for acceptance criteria, and if the acceptance criteria don't mention performance, the agent doesn't think about it. **Implication:** Non-functional requirements are not technical nice-to-haves. They are product requirements that determine whether the product stays usable over time. Features that ship fast but degrade are features that ship once. The Product Maker is the only role with the authority and incentive to mandate them. **Decision:** Every PRD includes non-functional requirements stated in human terms: performance budgets, security posture, resilience expectations, scale targets. The Product Maker defines the experience; the SWE designs the mechanisms. See `product-maker.md` "Non-Functional Requirements." | 2026-03-14 |
| 20 | Silent degradation is a trust violation | **Insight:** Four decisions in the Nacre 0.5 pipeline shared the same failure mode: a feature that worked but produced incomplete output without telling the user. A CSV parser that silently fell back to text ingestion when it couldn't parse European locale exports. Provenance badges that silently didn't appear when the model forgot annotation format. Image stitching that silently under-combined pages. OCR that silently capped at 10 pages. In every case the system ran without error. In every case the user's expectation wasn't met and they wouldn't discover it until later — when they asked a question the data should have answered, when they looked for a citation that wasn't there, when they referenced a document that was missing pages. **Implication:** Silent fallbacks that produce plausible-looking wrong or incomplete output are worse than errors. An error tells the user something went wrong. A plausible-looking partial result tells the user everything is fine. The user builds on it, references it, trusts it — and the trust violation compounds over time. This is the same structural problem as building AI products for children who can't consent: the worst failure isn't the system breaking, it's the system producing something that looks right but isn't, and the person relying on it has no way to tell. **Decision:** When a feature degrades, the user must know it degraded. If the system can't do what the user expects at full fidelity, it says so and offers the user a choice. This applies at the product level (degradation paths require user-visible indicators, not just logging), at the architecture level (fallback paths need notification mechanisms), and at the task level (acceptance criteria must include degradation-path tests, not just happy-path tests). | 2026-03-24 |
| 21 | Guardrails are defaults, not walls | **Insight:** Four decisions in the Nacre 0.5 pipeline implemented guardrails — reasonable defaults protecting against cost, privacy, or context window overload. An OCR cap at 10 pages. A 200K token document budget. Per-instance privacy permissions. Ask-every-time for ambiguous images. In every case the default was correct for the common case. In every case the problem was that the user couldn't override it. The user who drops a 180-page government filing and gets 10 pages didn't ask for cost control — they asked Nacre to read their document. The user asking for cross-document patterns gets funneled into a multi-step negotiation because the budget guardrail treats all queries the same way. **Implication:** A guardrail with no override is not a protection — it's a limitation the product chose to impose. The user who hits a guardrail and can't get past it experiences the product as fundamentally limited, not as making a smart tradeoff on their behalf. Guardrails should protect the common case (don't accidentally spend $50 on OCR for a casual upload) while respecting the user who knows what they want (process the whole document, I'll wait). **Decision:** Every guardrail must have a user-controlled escape valve. Defaults protect the common case. Overrides serve the user who knows what they want. If the override would be genuinely dangerous (data loss, irreversible action, significant cost), require explicit confirmation. Otherwise, let the user through. When designing a guardrail, specify both the default behavior and the override path. A guardrail without a documented override path fails the product review. | 2026-03-24 |
| 25 | PRDs for UI products require a First-Use Walkthrough | **Insight:** Nacre 0.5 shipped with four gaps discoverable in the first 5 minutes of user testing — no way to add a project from the UI, conversations displaying as UUIDs, no file watcher on new documents, and the system prompt not telling the model its own capabilities. The PRD specified these as system capabilities but never described how the user discovers, configures, or recovers from them. The peer review checked the PRD against the XRD but didn't walk the product as a user. The entire pipeline reviewed the spec structurally but never experientially. **Implication:** A PRD that describes what the system does but not what the user does in their first session produces a product that works but isn't usable. Structural review (contradictions, gaps between documents) catches a different class of error than experiential review (what happens when a person sits down and tries to use this). Both are required. Experiential review must be part of PRD creation so the product-agent thinks about user flows while writing, not after. **Decision:** UI products require a First-Use Walkthrough section in the PRD — a step-by-step description of the user's first session from launch through first meaningful action, including empty states, configuration discovery, and failure paths. The peer reviewer independently walks the user journey before reading the PRD and compares. This is a gate — PRDs without it don't advance. See `prompts/product-agent.md` and `prompts/peer-review-agent.md`. | 2026-03-24 |
| 22 | Don't make people re-authorize work they already asked for | **Insight:** Three patterns in the Nacre 0.5 pipeline and the build system itself shared the same root: the system treated a completed authorization as expired and asked again. The orchestrator ran Stages 1-8 and stopped before Stage 9, even though the human said "follow the build process" and the pipeline includes building. The product agent surfaced Tier 3 items to the human that it could have resolved itself — treating its own uncertainty as a reason to ask rather than a reason to think harder. Privacy permission asked every single time the user's private documents were relevant to a generated file, even within the same project where the user had already said yes. **Implication:** Re-authorization friction scales with engagement. The user who trusts the system more — uses it more, shares more private material, asks for more ambitious cross-document work — gets asked more questions. The system punishes depth. This fights the core value proposition of any product that promises to know the user over time. At the build system level, the same pattern creates a stop-and-wait default where the stated intent is keep-moving. The cost of one unnecessary stop is small. The cost of stopping by default at every ambiguity point is a system that requires the human to push it forward instead of pulling the human in only when it's stuck. **Decision:** The default is to continue. Stopping requires a reason — a gate failure, a blocking decision that genuinely requires human judgment, a constraint the system cannot resolve. Re-asking requires a reason — the context changed, the scope changed, the prior authorization doesn't cover the current situation. "I'm not sure" is not a reason to stop. It's a reason to reframe, re-evaluate, and attempt resolution before escalating. This applies to the build system (the orchestrator advances unless stopped), to agents (attempt a position before escalating), to tier evaluation (reframe through insight/implication before classifying as Tier 3), and to products (don't ask permission the user already granted in the same context). | 2026-03-24 |

## Operational Decisions — How to build well

Principles about how work gets done — the pipeline, the roles, the quality bar, the decision-making process itself. They apply to every project built under `~/building/`. An agent running the pipeline should follow these.

| # | Decision | Rationale | Date |
|---|----------|-----------|------|
| 1 | Decisions are final once documented | Relitigating resolved questions is the primary source of wasted time in both product thinking and code. The Decisions Log pattern enforces finality. Reopen only with new material information, not new feelings. | 2026-03-08 |
| 2 | Tests define done | "It works on my machine" and "it compiles" are not done. A test that fails when the feature is broken and passes when it works — that's done. Every task has acceptance criteria expressed as tests. | 2026-03-08 |
| 3 | No dependencies without justification | Every external dependency is a liability: version conflicts, security vulnerabilities, behavior changes, abandoned projects. Native code that does the job in a reasonable number of lines wins over a package. If a dependency is necessary, document why in the task file. | 2026-03-08 |
| 4 | Documents before code | The pipeline (PRD → XRD → Peer Review → Test Plan → Tasks → Build) is not optional. Steps can be fast — a PRD can be written in an evening, an XRD in an hour — but they cannot be skipped. Each stage catches errors the next stage would inherit. | 2026-03-08 |
| 6 | Files over conversations | Anything that needs to survive a session lives in a file. Conversations are for thinking. Files are for building. Context that exists only in chat history is context that's already lost. | 2026-03-08 |
| 7 | One task, one change | Scope discipline at the task level. Each task produces a reviewable, testable unit of work. Opportunistic fixes go on a separate task list, not into the current task. | 2026-03-08 |
| 9 | Test plan source of truth is the PRD | The test plan is written from the PRD as the source of truth for product intent. The XRD is consulted for implementation-revealed edge cases but does not override PRD-driven coverage. Any test case sourced from the XRD rather than the PRD is tagged [XRD] to distinguish implementation discovery from product intent. | 2026-03-08 |
| 10 | Roles resolve issues with each other before escalating to the human | The human is the boss, not the bus. SWE asks Product to resolve PRD ambiguity before escalating. Tester routes contradictions to the claim owner first. SWE implementation choices include tradeoff statements (user impact, maintenance burden, refactoring risk). Escalation to the human happens when roles disagree on materiality, refactoring risk is non-trivial, or tradeoffs cross project boundaries. | 2026-03-09 |
| 28 | Stage 10 uses Playwright MCP for automated smoke testing | **Insight:** Stage 10 was designed as a three-level progression: Level 1 (manual checklist), Level 2 (agent-driven with browser), Level 3 (continuous). Playwright MCP is now installed globally for Claude Code, giving every session the ability to drive a real browser. The progression from Level 1 to Level 2 is immediate — no custom tooling needed. The smoke test agent uses the same Playwright MCP tools available to any Claude Code session. **Implication:** The smoke test no longer requires the human to execute the walkthrough manually. The agent opens the browser, performs each walkthrough step, captures evidence, and reports pass/fail. The human's role shifts from executing the test to starting the server and reviewing failures. This closes the gap that caused Nacre 0.5 through 0.6 to ship with broken user experiences — every milestone now gets verified against the running product by an agent that can see what the user sees. **Decision:** Stage 10 uses Playwright MCP for UI products. The orchestrator executes the PRD's First-Use Walkthrough against the running product, captures screenshots as evidence, and reports pass/fail per step. Failed steps become fix tasks. The milestone is not complete until all walkthrough steps pass. The human starts the server and reviews failures. Everything else is automated. See `orchestrator.md` Stage 10 and `prompts/smoke-test-protocol.md`. | 2026-03-25 |
| 27 | Briefs are decomposed into milestones before they are decomposed into tasks | **Insight:** Nacre went through four build cycles (0.5, 0.5.1, wiring fix, 0.6). Every cycle passed all tests. Every cycle failed when a user sat down with the product. Two root causes recurred: integration bugs that no individual task's tests could catch (a 3-line wiring omission, a config constant used as a context window instead of an output limit), and milestone scope so large that these bugs weren't discovered until everything was built. A 3-line wiring fix and a config correction would have been caught in minutes if the first extractor's output had been smoke tested against the running product before the remaining extractors were built. **Implication:** The pipeline treats a brief as a single unit: one PRD, one XRD, one build, one smoke test. Integration problems compound across the entire brief's scope before anyone checks whether the product works. Smaller milestones with per-milestone smoke tests surface integration problems at the first milestone, when the blast radius is smallest and the fix is cheapest. **Decision:** Briefs are decomposed into milestones before they are decomposed into tasks. Each milestone produces working software the user can touch. Each milestone gets its own pipeline pass (PRD through smoke test). The SDM reassesses after each milestone. The next milestone does not begin until the current milestone's smoke test passes against the running product. See `orchestrator.md` Stage 0 and `build-process.md` Stage 0. | 2026-03-25 |
| 26 | The pipeline includes a Smoke Test stage after Build | **Insight:** Nacre 0.5.1 ran the full pipeline — PRD with 18 decisions, XRD with 12 decisions, peer review that caught 8 issues, test plan with 91 cases, 11 tasks executed, 1383 tests passing. The product still didn't work: the model told users it couldn't see their documents while listing the documents it could see. Every test verified code against the spec. No test verified the product against the user. **Implication:** Unit tests, integration tests, and acceptance criteria verify that code does what the spec says. They do not verify that the product does what the user expects. A product can pass every test and still be broken when the spec was correct but incomplete, when the code implements the spec but runtime behavior differs, or when the integration between components produces emergent behavior no individual test covers. The pipeline needs a stage that checks the product as a whole, from the user's perspective, after all code is complete. **Decision:** The pipeline includes a Smoke Test stage after Build. The PRD's First-Use Walkthrough (written before code exists) becomes a verification checklist (executed against the running product after code ships). Every walkthrough step must pass against the running product for the milestone to be complete. Failed steps become bug reports that feed back into the build. See `orchestrator.md` Stage 10 and `build-process.md` Stage 8. | 2026-03-24 |
| 16 | Comprehension before execution, at every level | **Insight:** Two independent feedback sources — a user applying the framework to an existing production codebase, and Yegge/Kim's three-loop model in *Vibe Coding* — converged on the same structural gap. The framework prescribes comprehension at the project level (XRD responds to PRD) but not at the idea-intake level or the task level. The Product Maker accepted a thin idea brief at face value and produced a PRD that echoed the vagueness. Agents read task files and started coding without stating what they understood. The framework had no inner loop — no prescribed step between "read the input" and "produce the output" where misunderstandings surface. **Implication:** Misalignment compounds. A shallow intake produces a shallow PRD. A misunderstood task produces code that passes its own tests but misses the intent. The cheapest place to catch misunderstanding is before work begins, at every level: before writing the PRD, and before writing code for each task. **Decision:** (1) The Product Maker must play back understanding of the idea brief and get human confirmation before writing the PRD (see `product-maker.md` "Play Back Your Understanding Before Writing"). (2) Agents must fill in an Execution Plan section in each task file before writing code (see `task-template.md`). (3) The framework now includes guidance for existing codebases: guardrails in the project CLAUDE.md, codebase discovery before the PRD, deployment-aware task decomposition (see `build-process.md` "Working Against an Existing Codebase"). | 2026-03-16 |

## Build Process Mechanics — How the machinery works

Specific rules about the build process machinery — task templates, kill protocols, gate enforcement, loop boundaries. Implementation details of the operational principles. An agent executing a task should follow these.

| # | Decision | Rationale | Date |
|---|----------|-----------|------|
| 5 | Decompose for concurrency, build sequentially until you trust the process | Every build plan is structured as parallel tracks with Day Zero contracts. Actual execution is sequential until you've completed at least one full track and understand the agent feedback loop well enough to run two tracks without losing quality. | 2026-03-08 |
| 11 | Rejected ideas have exactly two types: Hard Kill and Deferred | Hard Kill: idea is permanently dead, triggers immediate purge of all references across all documents and code. Deferred: idea is valid but out of scope for now, surviving references are tagged with the deferral decision number. No other rejection types exist — the reason for killing an idea belongs in the rationale column, not the type. | 2026-03-12 |
| 12 | Hard Kill decisions trigger an immediate cross-document purge | When a Hard Kill is logged, search all project documents and source code for the killed idea's terminology before doing anything else. Delete every reference that is not the DECISIONS.md kill entry itself. This is the one exception to one-task-one-change: purging a ghost takes priority over scope discipline because an unremoved ghost compounds with every subsequent session. Tag all Hard Kill entries `[HARD KILL]`. Tag all Deferred entries `[DEFERRED]`. | 2026-03-12 |
| 13 | Quality bar examples are architecture test cases, not inspiration | **Insight:** On a project, smoke test examples existed before any code was written — hand-built reference outputs showing the depth of analysis the tool should produce. The build process produced 20 tasks and a complete architecture that could not reproduce those examples, because intermediate processing layers discarded the data relationships the examples depended on. Every component passed its tests. The output missed the quality bar by a wide margin. The gap was structural, not incremental — the architecture made it impossible to hit the quality bar. **Implication:** The build process had no step that validated "can this architecture reproduce the quality bar example?" The SWE decomposed the output's structure into components without studying the process that created it. The architecture mirrored what the output looked like, not how it was made. The Peer Review had no protocol for checking output reproducibility. Nobody compared the final output to the reference examples until after 20 tasks were complete. **Decision:** Before task decomposition, the SWE must trace at least one quality bar example through the proposed architecture (Quality Bar Trace in `swe.md`). The Peer Review must verify this trace (`peer-reviewer.md`). Cost-quality tradeoffs in the architecture are surfaced to the Product Maker, not silently resolved by the SWE. At least one task must include acceptance criteria that reference the quality bar directly. | 2026-03-14 |
| 14 | Architect the creation process, not the output structure | **Insight:** An architecture had ten analyzers mapping to ten categories visible in the reference output (consumables, rotation, deaths, gear, etc.). The reference output was not *produced* by categories — it was produced by reading raw chronological data and letting patterns emerge. The architecture mirrored the output's table of contents, not the process that created it. This felt like understanding but was classification. **Implication:** When the SWE decomposes by output structure, the architecture can only find patterns the designer anticipated. When the creation process involves emergent pattern discovery, an architecture that pre-filters by category eliminates exactly the signal the final stage needs. **Decision:** Before proposing an architecture, the SWE must study how the quality bar output was created: what the creator needed to see simultaneously, what reasoning connected input to output, what process (not structure) produced the insight. The insight/implication format is the interface between loops — inner loop insights feed middle loop synthesis. Missing insights break the signal chain. **Decision:** A task's status cannot change to `done` without a Completed section containing date, deviations, and insight/implication. A task without a Completed section is `in progress` regardless of test status. | 2026-03-20 |

## Decision Taxonomy

Every decision has at minimum a rationale (the tables above). When a decision emerged from a learning — a build session, a peer review, a process failure — use the enriched taxonomy:

- **Insight:** What we learned. The observation that changed our understanding. Not weather reporting ("this was hard") — a structural observation that reveals something non-obvious.
- **Implication:** What that means for how we work. The consequence of the insight applied forward.
- **Decision:** What we chose to do about it, stated as a closed, actionable commitment.

The enriched taxonomy is optional. Simple resolved questions ("use SQLite, not CoreData") need only the rationale column. Decisions that emerged from pattern recognition, failure analysis, or cross-project learning should use all three fields so the reasoning compounds across sessions.

When using the enriched taxonomy in the table, the Rationale cell contains all three fields separated by line breaks. When documenting outside the table (in task completion notes, peer reviews, or process docs), use the standalone format:

> **Insight:** [what we learned]
> **Implication:** [what that means]
> **Decision:** [what we chose]

## Reopen Protocol

To reopen a closed decision:

1. State the original decision number and text.
2. State the new information that was not available when the decision was made.
3. State why that information is material — why it changes the calculus, not just adds a data point.

The decision is only reopened if the Product Maker (me) affirms the information is material. If reopened, the original entry stays in the table with an annotation: `[Reopened DATE — see #N]`. A new entry is added with the updated decision and rationale. The log is append-only. Nothing is deleted or overwritten.

**Examples of what qualifies as new material information:**
- "Decision #3 said no dependencies. During Task 12 I discovered that Apple's native NLP tokenizer doesn't support the language we need. The only option is a third-party library." — The constraint didn't exist when the decision was made. Material.
- "Decision #5 said build sequentially. We've now completed three full tracks across two projects without quality loss. The sequential constraint is no longer earning its keep." — Empirical evidence from running the process. Material.

**Examples of what does not:**
- "Decision #8 says abstractions must be earned, but building without them feels slow." — Friction is not new information. The decision already accounts for the tradeoff.
- "I found a cleaner way to organize the persistence layer than what the XRD specified." — Preference is not new information. If the current approach works and tests pass, the decision stands.
- "Another project uses a different pattern for this." — External precedent is not material unless it reveals a constraint this project faces.

## Deferred Topics

These were identified in the first peer review and are consciously deferred to the project level. They are not in scope for the master build docs. When they surface during a project, resolve them in that project's DECISIONS.md.

- **Error handling philosophy.** What does an agent do when a task is ambiguous, a test is flaky, or two tasks produce conflicting outputs? "Ask before guessing" covers code ambiguity. Process failures are project-specific.
- **Session handoff protocol.** What state should the codebase be in when a session ends mid-task? The task file status field (`in progress`) tracks this, but guidance on clean stopping points is project-dependent.
- **When to scale down stages.** The system says "no step is skipped" but doesn't define minimal-viable versions of each stage for a 2-hour project vs. a multi-week build. Scaling guidance depends on project scope and will be documented per-project as patterns emerge.
