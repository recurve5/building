# Decisions

Cross-project principles. These apply to everything built under `~/building/`. They are closed unless new material information surfaces.

| # | Decision | Rationale | Date |
|---|----------|-----------|------|
| 1 | Decisions are final once documented | Relitigating resolved questions is the primary source of wasted time in both product thinking and code. The Decisions Log pattern enforces finality. Reopen only with new material information, not new feelings. | 2026-03-08 |
| 2 | Tests define done | "It works on my machine" and "it compiles" are not done. A test that fails when the feature is broken and passes when it works — that's done. Every task has acceptance criteria expressed as tests. | 2026-03-08 |
| 3 | No dependencies without justification | Every external dependency is a liability: version conflicts, security vulnerabilities, behavior changes, abandoned projects. Native code that does the job in a reasonable number of lines wins over a package. If a dependency is necessary, document why in the task file. | 2026-03-08 |
| 4 | Documents before code | The pipeline (PRD → XRD → Peer Review → Test Plan → Tasks → Build) is not optional. Steps can be fast — a PRD can be written in an evening, an XRD in an hour — but they cannot be skipped. Each stage catches errors the next stage would inherit. | 2026-03-08 |
| 5 | Decompose for concurrency, build sequentially until you trust the process | Every build plan is structured as parallel tracks with Day Zero contracts. Actual execution is sequential until you've completed at least one full track and understand the agent feedback loop well enough to run two tracks without losing quality. | 2026-03-08 |
| 6 | Files over conversations | Anything that needs to survive a session lives in a file. Conversations are for thinking. Files are for building. Context that exists only in chat history is context that's already lost. | 2026-03-08 |
| 7 | One task, one change | Scope discipline at the task level. Each task produces a reviewable, testable unit of work. Opportunistic fixes go on a separate task list, not into the current task. | 2026-03-08 |
| 8 | Abstractions must be earned | Every abstraction is a bet on future change. For v1 of any project, the default is fewer abstractions. An abstraction earns its place when the cost of not having it is concrete and near-term. "We might need this later" is not earned. | 2026-03-08 |
| 9 | Test plan source of truth is the PRD | The test plan is written from the PRD as the source of truth for product intent. The XRD is consulted for implementation-revealed edge cases but does not override PRD-driven coverage. Any test case sourced from the XRD rather than the PRD is tagged [XRD] to distinguish implementation discovery from product intent. | 2026-03-08 |
| 10 | Roles resolve issues with each other before escalating to the human | The human is the boss, not the bus. SWE asks Product to resolve PRD ambiguity before escalating. Tester routes contradictions to the claim owner first. SWE implementation choices include tradeoff statements (user impact, maintenance burden, refactoring risk). Escalation to the human happens when roles disagree on materiality, refactoring risk is non-trivial, or tradeoffs cross project boundaries. | 2026-03-09 |
| 11 | Rejected ideas have exactly two types: Hard Kill and Deferred | Hard Kill: idea is permanently dead, triggers immediate purge of all references across all documents and code. Deferred: idea is valid but out of scope for now, surviving references are tagged with the deferral decision number. No other rejection types exist — the reason for killing an idea belongs in the rationale column, not the type. | 2026-03-12 |
| 12 | Hard Kill decisions trigger an immediate cross-document purge | When a Hard Kill is logged, search all project documents and source code for the killed idea's terminology before doing anything else. Delete every reference that is not the DECISIONS.md kill entry itself. This is the one exception to one-task-one-change: purging a ghost takes priority over scope discipline because an unremoved ghost compounds with every subsequent session. Tag all Hard Kill entries `[HARD KILL]`. Tag all Deferred entries `[DEFERRED]`. | 2026-03-12 |
| 13 | Quality bar examples are architecture test cases, not inspiration | **Insight:** On a project, smoke test examples existed before any code was written — hand-built reference outputs showing the depth of analysis the tool should produce. The build process produced 20 tasks and a complete architecture that could not reproduce those examples, because intermediate processing layers discarded the data relationships the examples depended on. Every component passed its tests. The output missed the quality bar by a wide margin. The gap was structural, not incremental — the architecture made it impossible to hit the quality bar. **Implication:** The build process had no step that validated "can this architecture reproduce the quality bar example?" The SWE decomposed the output's structure into components without studying the process that created it. The architecture mirrored what the output looked like, not how it was made. The Peer Review had no protocol for checking output reproducibility. Nobody compared the final output to the reference examples until after 20 tasks were complete. **Decision:** Before task decomposition, the SWE must trace at least one quality bar example through the proposed architecture (Quality Bar Trace in `swe.md`). The Peer Review must verify this trace (`peer-reviewer.md`). Cost-quality tradeoffs in the architecture are surfaced to the Product Maker, not silently resolved by the SWE. At least one task must include acceptance criteria that reference the quality bar directly. | 2026-03-14 |
| 14 | Architect the creation process, not the output structure | **Insight:** An architecture had ten analyzers mapping to ten categories visible in the reference output (consumables, rotation, deaths, gear, etc.). The reference output was not *produced* by categories — it was produced by reading raw chronological data and letting patterns emerge. The architecture mirrored the output's table of contents, not the process that created it. This felt like understanding but was classification. **Implication:** When the SWE decomposes by output structure, the architecture can only find patterns the designer anticipated. When the creation process involves emergent pattern discovery, an architecture that pre-filters by category eliminates exactly the signal the final stage needs. **Decision:** Before proposing an architecture, the SWE must study how the quality bar output was created: what the creator needed to see simultaneously, what reasoning connected input to output, what process (not structure) produced the insight. The architecture recapitulates the creation process. See `swe.md` "Understand the Problem Before Architecting It." | 2026-03-14 |
| 15 | Non-functional requirements are product requirements | **Insight:** Agents build features that work correctly but ignore performance, security, and resilience unless explicitly told. No pagination, no indexing, no caching — the app works on day one and degrades from there. The agent optimizes for acceptance criteria, and if the acceptance criteria don't mention performance, the agent doesn't think about it. **Implication:** Non-functional requirements are not technical nice-to-haves. They are product requirements that determine whether the product stays usable over time. Features that ship fast but degrade are features that ship once. The Product Maker is the only role with the authority and incentive to mandate them. **Decision:** Every PRD includes non-functional requirements stated in human terms: performance budgets, security posture, resilience expectations, scale targets. The Product Maker defines the experience; the SWE designs the mechanisms. See `product-maker.md` "Non-Functional Requirements." | 2026-03-14 |
| 16 | Comprehension before execution, at every level | **Insight:** Two independent feedback sources — a user applying the framework to an existing production codebase, and Yegge/Kim's three-loop model in *Vibe Coding* — converged on the same structural gap. The framework prescribes comprehension at the project level (XRD responds to PRD) but not at the idea-intake level or the task level. The Product Maker accepted a thin idea brief at face value and produced a PRD that echoed the vagueness. Agents read task files and started coding without stating what they understood. The framework had no inner loop — no prescribed step between "read the input" and "produce the output" where misunderstandings surface. **Implication:** Misalignment compounds. A shallow intake produces a shallow PRD. A misunderstood task produces code that passes its own tests but misses the intent. The cheapest place to catch misunderstanding is before work begins, at every level: before writing the PRD, and before writing code for each task. **Decision:** (1) The Product Maker must play back understanding of the idea brief and get human confirmation before writing the PRD (see `product-maker.md` "Play Back Your Understanding Before Writing"). (2) Agents must fill in an Execution Plan section in each task file before writing code (see `task-template.md`). (3) The framework now includes guidance for existing codebases: guardrails in the project CLAUDE.md, codebase discovery before the PRD, deployment-aware task decomposition (see `build-process.md` "Working Against an Existing Codebase"). | 2026-03-16 |

## Decision Taxonomy

Every decision has at minimum a rationale (the table above). When a decision emerged from a learning — a build session, a peer review, a process failure — use the enriched taxonomy:

- **Insight:** What we learned. The observation that changed our understanding. Not weather reporting ("this was hard") — a structural observation that reveals something non-obvious.
- **Implication:** What that means for how we work. The consequence of the insight applied forward.
- **Decision:** What we chose to do about it, stated as a closed, actionable commitment.

The enriched taxonomy is optional. Simple resolved questions ("use SQLite, not CoreData") need only the rationale column. Decisions that emerged from pattern recognition, failure analysis, or cross-project learning should use all three fields so the reasoning compounds across sessions.

When using the enriched taxonomy in the table, the Rationale cell contains all three fields separated by line breaks. When documenting outside the table (in task completion notes, peer reviews, or process docs), use the standalone format:

> **Insight:** [what we learned]
> **Implication:** [what that means]
> **Decision:** [what we chose]

## Reopen Protocol

To reopen a closed decision:

1. State the original decision number and text.
2. State the new information that was not available when the decision was made.
3. State why that information is material — why it changes the calculus, not just adds a data point.

The decision is only reopened if the Product Maker (me) affirms the information is material. If reopened, the original entry stays in the table with an annotation: `[Reopened DATE — see #N]`. A new entry is added with the updated decision and rationale. The log is append-only. Nothing is deleted or overwritten.

**Examples of what qualifies as new material information:**
- "Decision #3 said no dependencies. During Task 12 I discovered that Apple's native NLP tokenizer doesn't support the language we need. The only option is a third-party library." — The constraint didn't exist when the decision was made. Material.
- "Decision #5 said build sequentially. We've now completed three full tracks across two projects without quality loss. The sequential constraint is no longer earning its keep." — Empirical evidence from running the process. Material.

**Examples of what does not:**
- "Decision #8 says abstractions must be earned, but building without them feels slow." — Friction is not new information. The decision already accounts for the tradeoff.
- "I found a cleaner way to organize the persistence layer than what the XRD specified." — Preference is not new information. If the current approach works and tests pass, the decision stands.
- "Another project uses a different pattern for this." — External precedent is not material unless it reveals a constraint this project faces.

## Deferred Topics

These were identified in the first peer review and are consciously deferred to the project level. They are not in scope for the master build docs. When they surface during a project, resolve them in that project's DECISIONS.md.

- **Error handling philosophy.** What does an agent do when a task is ambiguous, a test is flaky, or two tasks produce conflicting outputs? "Ask before guessing" covers code ambiguity. Process failures are project-specific.
- **Session handoff protocol.** What state should the codebase be in when a session ends mid-task? The task file status field (`in progress`) tracks this, but guidance on clean stopping points is project-dependent.
- **When to scale down stages.** The system says "no step is skipped" but doesn't define minimal-viable versions of each stage for a 2-hour project vs. a multi-week build. Scaling guidance depends on project scope and will be documented per-project as patterns emerge.
